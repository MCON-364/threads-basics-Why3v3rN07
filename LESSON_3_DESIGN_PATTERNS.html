When should we use threads?

Threads are useful when a program has work that can overlap safely. They are not magic speed buttons; they add complexity and must be justified.

⏳

Use threads when work waits

Examples: waiting for files, network calls, database queries, user input, or a slow external service. While one task waits, another can run.

🧮

Use threads for parallel computation

Examples: splitting a large calculation into independent chunks. This helps only when the work is CPU-heavy enough to overcome thread overhead.

🖥️

Use threads for responsiveness

Examples: UI thread remains responsive while background work loads data, saves files, or computes a report.

Do not use threads just because code is slow. First ask whether the work can safely happen independently. If everything depends on the previous step, threads may only add bugs.
QuestionIf yesIf no
Can the work be divided into independent tasks?Threads or executors may help.Keep it sequential or redesign the algorithm.
Does the program spend time waiting?Threads can improve throughput and responsiveness.Look at CPU cost and algorithmic complexity first.
Is there shared mutable state?You need a synchronization plan.The design is much safer and simpler.
Do you need results back?Consider Callable and Future.Runnable or fire-and-forget task submission may be enough.

Pattern 1: Producer-consumer

This is the most important pattern for connecting concurrency back to data structures. The queue is the shared structure, and it must be thread-safe.

Shape of the pattern

Producer
creates tasks
→
BlockingQueue
safe handoff
→
Consumer
processes tasks
Key idea: do not manually synchronize an ArrayList as a work queue. Use a queue designed for concurrent handoff.

Starter example

BlockingQueue<Task> queue = new LinkedBlockingQueue<>();

Runnable producer = () -> {
    queue.put(new Task("email report"));
};

Runnable consumer = () -> {
    Task task = queue.take();
    process(task);
};
A blocking queue coordinates waiting for you: consumers can wait when the queue is empty, and bounded queues can slow producers when the queue is full.

Producer-Consumer Data Structure Choices

The producer-consumer pattern is built around one central question: how should work move from the thread that creates it to the thread that processes it? The answer depends on ordering, capacity, waiting behavior, and whether work should be buffered or handed off directly.

Core idea: the queue is not just a storage container. In producer-consumer programs, the queue becomes the communication boundary between threads.

Basic producer-consumer flow

Step 1 — Decide what the queue must guarantee

Before picking a Java class, describe the behavior you need in plain English. Most producer-consumer bugs come from choosing a data structure before deciding the contract.

Design question Why it matters Likely data structure
Should the producer slow down when consumers fall behind? This is back pressure. Without it, work can pile up until memory runs out. ArrayBlockingQueue or bounded LinkedBlockingQueue
Does work need to be processed in arrival order? FIFO ordering is the normal default for logs, messages, uploaded files, and ordinary tasks. ArrayBlockingQueue or LinkedBlockingQueue
Does the most important work go first? Priority scheduling ignores arrival order and chooses based on a ranking rule. PriorityBlockingQueue
Should the producer hand work directly to a waiting consumer? Zero buffering gives direct handoff, but the producer and consumer become tightly coupled. SynchronousQueue holds no elements at all — a call to put() blocks until another thread calls take(), so every insertion is a synchronous rendezvous between two threads. SynchronousQueue
Should work become available only after a delay? Useful for retries, cache expiration, scheduled cleanup, and timeout handling. DelayQueue wraps each element in a delay contract — elements must implement the Delayed interface, and take() will block until the element with the shortest remaining delay has actually expired. DelayQueue
Do I only need a queue inside one thread? No inter-thread handoff means no blocking queue is needed. ArrayDeque or PriorityQueue
Use case A — ordinary task pipeline: bounded FIFO queue

This is the most common producer-consumer design. Producers create tasks. Consumers process them. Tasks should usually be handled in the order they arrive, and the system should not accept unlimited work.

Best fit

  • ArrayBlockingQueue when you want a fixed-size buffer and predictable memory.
  • Bounded LinkedBlockingQueue when you want FIFO behavior with a capacity and potentially better producer/consumer concurrency.
  • Use put and take when waiting is the correct behavior.
Teaching shortcut: start here unless the problem clearly requires priority, delay, or direct handoff.

Example

BlockingQueue<Task> queue =
    new ArrayBlockingQueue<>(100);

// Producer
queue.put(task);     // waits if buffer is full

// Consumer
Task task = queue.take(); // waits if buffer is empty
process(task);
Use case B — urgent work first: priority queue

Sometimes FIFO is the wrong rule. A hospital triage system, build server, print queue, or task manager may need to process high-priority work before older low-priority work.

Best fit

  • Use PriorityBlockingQueue when producers and consumers run on different threads.
  • Use PriorityQueue only in single-threaded code or with external synchronization.
  • Define priority with Comparable or a Comparator.
Important: PriorityBlockingQueue is unbounded. put will not block because of capacity. If producers can overwhelm memory, add your own admission control or use a different design.

Example

BlockingQueue<Task> queue =
    new PriorityBlockingQueue<>(
        11,
        Comparator.comparing(Task::priority)
    );

queue.put(new Task("routine", LOW));
queue.put(new Task("urgent", HIGH));

Task next = queue.take(); // urgent first
Use case C — direct handoff: no buffer

In some programs, storing work in a queue is not desired. The producer should only continue when a consumer is ready to accept the task. This creates a direct handoff.

Best fit

  • Use SynchronousQueue when the queue should have zero capacity.
  • Each producer operation must meet a consumer operation.
  • This design provides strong back pressure because nothing can pile up.
Mental model: not a mailbox, but a handshake.

Example

BlockingQueue<Task> handoff =
    new SynchronousQueue<>();

// Producer blocks until a consumer is ready
handoff.put(task);

// Consumer blocks until a producer provides work
Task task = handoff.take();
Use case D — retry later or scheduled work: delayed queue

Some work should not be processed immediately. For example, retry a failed request after 30 seconds, expire a cache entry after 10 minutes, or run cleanup after a timeout.

Best fit

  • Use DelayQueue for tasks that become available only after a delay expires.
  • Elements must implement Delayed.
  • take blocks until the earliest delayed item is ready.

Example idea

DelayQueue<RetryTask> retries =
    new DelayQueue<>();

retries.put(new RetryTask(request, 30));

// waits until delay expires
RetryTask task = retries.take();
retry(task);
Use case E — non-blocking shared work list: concurrent queue

Sometimes you want a thread-safe queue, but you do not want producers or consumers to block. This is useful for event collection, logging buffers, or best-effort work queues where a thread can check for work and move on if none exists.

Best fit

  • Use ConcurrentLinkedQueue for lock-free FIFO access.
  • poll returns null immediately if empty.
  • There is no built-in back pressure because it is unbounded and non-blocking.
Watch out: non-blocking does not mean simpler. If consumers keep polling in a tight loop, they can waste CPU.

Example

Queue<Event> events =
    new ConcurrentLinkedQueue<>();

// Producer
events.offer(event);

// Consumer
Event event = events.poll();
if (event != null) {
    handle(event);
}

Shutdown choices in producer-consumer programs

Picking the queue is only half the design. You also need to decide how consumers know when to stop.

☠️

Poison pill

Producer inserts a special value such as DONE. Consumers stop when they take it.

Best when: the number of consumers is known.

⏱️

Timed poll

Consumer calls poll(timeout, unit) and periodically checks a shutdown flag.

Best when: consumers need graceful shutdown without special queue values.

⚡

Interrupt

Another thread interrupts a blocked consumer. The worker restores the interrupt flag and exits.

Best when: immediate cancellation is needed.

Interruption contract

try {
    Task task = queue.take();
    process(task);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // restore flag
    // clean up and exit the worker loop
}
Final decision rule: for a normal producer-consumer program, start with a bounded BlockingQueue. Move away from it only when the requirements clearly say priority ordering, delayed availability, direct handoff, or non-blocking polling.

Pattern 2: Worker Pool (ExecutorService)

The worker pool pattern separates task submission from task execution. Instead of creating threads manually, we submit work to a pool that manages a fixed set of worker threads.

Conceptual model

Tasks wait in a queue → worker threads pull tasks → pool controls concurrency.
❌

Manual thread creation (anti-pattern)

for (Task task : tasks) {
    new Thread(() -> process(task)).start();
}
  • Unbounded thread creation → resource exhaustion
  • No coordination or backpressure
  • Difficult shutdown
  • No result tracking
✅

Worker pool (recommended)

ExecutorService pool =
    Executors.newFixedThreadPool(4);

for (Task task : tasks) {
    pool.submit(() -> process(task));
}

pool.shutdown();
  • Controls max concurrency
  • Reuses threads (efficient)
  • Supports results via Future
  • Centralized lifecycle

Common use cases

🌐

Web server

Each request is a task. Thread pool limits concurrent requests to prevent overload.

📂

Batch processing

Process files or records in parallel while controlling CPU usage.

🔄

Background jobs

Queue jobs (emails, reports) and let workers process asynchronously.

Design decisions: choosing a pool

GoalExecutorWhy
Limit CPU usage newFixedThreadPool(n) Caps number of concurrent threads
Handle bursts of short tasks newCachedThreadPool() Creates threads as needed, reuses idle ones
Scheduled work ScheduledExecutorService Runs tasks periodically or with delay
Single-threaded execution newSingleThreadExecutor() Guarantees sequential execution

→

Using Callable + Future

Future<Integer> f =
    pool.submit(() -> compute());

Integer result = f.get();
  • submit() returns Future
  • get() blocks until result ready

Common mistake

// BAD: kills concurrency
for (...) {
    Future f = pool.submit(...);
    f.get(); // blocks immediately
}
Collect futures first, then call get() later to keep execution parallel.

When to use a worker pool

Many independent tasks → Use worker pool
Need to limit concurrency → Fixed thread pool
Need results → Callable + Future
Need scheduling → ScheduledExecutorService
Key takeaway: Threads are workers, the pool is the manager, and tasks are the jobs. You design at the level of tasks—not threads.

Pattern 3: Pipeline Processing

A pipeline breaks work into sequential stages, where each stage performs a specific transformation. Stages run concurrently and communicate via queues, allowing work to flow through the system.

Conceptual model

Each stage runs independently. Queues decouple stages and allow overlap.

Example: Data Import Pipeline

Read rows
I/O bound
→
Validate
CPU / rules
→
Save
DB / I/O

Pipeline implementation (simplified)

BlockingQueue raw = new LinkedBlockingQueue<>();
BlockingQueue valid = new LinkedBlockingQueue<>();

// Stage 1: Reader
new Thread(() -> {
    for (Row r : readRows()) {
        raw.put(r);
    }
}).start();

// Stage 2: Validator
new Thread(() -> {
    while (true) {
        Row r = raw.take();
        if (isValid(r)) valid.put(r);
    }
}).start();

// Stage 3: Saver
new Thread(() -> {
    while (true) {
        Row r = valid.take();
        save(r);
    }
}).start();

What makes this a pipeline?

  • Each stage has a single responsibility
  • Data flows through queues
  • Stages run concurrently
  • No shared mutable state between stages

Execution timeline (overlap)

Notice how stages overlap—this is where performance gains come from.

When pipelines are the right choice

📥

Data ingestion

Read → clean → store large datasets efficiently.

🖼️

Image processing

Load → transform → save images in parallel stages.

📡

Streaming systems

Receive → process → forward events (Kafka-style).

Design decisions in a pipeline

QuestionDecision
How many threads per stage? CPU-bound → fewer threads; I/O-bound → more threads
Which queue? Usually BlockingQueue for safe handoff
Bounded or unbounded? Bounded queues prevent memory blow-up
How to stop? Poison pill or interruption strategy

Bottleneck detection

If one stage is slower than others, the queue before it will grow.

  • Growing queue → downstream bottleneck
  • Empty queue → upstream bottleneck
Pipelines make performance problems visible—but you must monitor them.
✅

Benefits

  • Stages execute in parallel
  • Clear separation of concerns
  • Scales by tuning individual stages
  • Easy to test each stage independently
⚠️

Costs

  • More complex architecture
  • Need coordinated shutdown
  • Error handling across stages is harder
  • Maintaining ordering can be tricky

When NOT to use a pipeline

Single simple task → Just use a method call
No natural stages → Use worker pool instead
Strong ordering requirements → Pipeline may complicate logic
Key takeaway: Pipelines are about flow. Each stage does one thing well, and queues connect them safely.

Decision process: how to design the program

Use this checklist before writing concurrency code. It helps prevent the common mistake of adding threads first and asking about correctness later.

  • Step 1: Describe the work. What are the tasks? Are they independent? Do they return results?
  • Step 2: Identify the bottleneck. Is the program waiting on I/O, using CPU, or blocked by one shared resource?
  • Step 3: Choose the concurrency shape. Sequential, stream pipeline, producer-consumer, worker pool, or pipeline.
  • Step 4: Mark shared mutable state. Lists, maps, counters, queues, caches, files, and database connections.
  • Step 5: Choose synchronization. Atomic value, lock, concurrent collection, blocking queue, latch, barrier, or semaphore.
  • Step 6: Plan lifecycle. How do threads start, stop, handle exceptions, and report results?
Classroom rule: every threaded design should be explainable in one sentence: “I am using ___ because ___, and I protect ___ using ___.”

Where and how should we synchronize?

Synchronization should protect correctness without blocking unrelated work. The goal is not “lock everything.” The goal is “protect exactly what must not be interleaved.”

Shared state shapeGood synchronization choiceWhy
One counter or flagAtomicInteger, AtomicBooleanSmall independent value; no need for a large lock.
Multiple fields must update togethersynchronized or ReentrantLockThe invariant spans more than one value.
Shared map with many reads/writesConcurrentHashMapDesigned for concurrent access without one giant object lock.
Producer-consumer queueBlockingQueueCombines thread-safe storage with waiting behavior.
Read-heavy cacheReentrantReadWriteLockAllows many simultaneous readers while writers remain exclusive.
Limit access to a few resourcesSemaphoreModels permits rather than exclusive ownership.

Keep the critical section small

lock.lock();
try {
    // only the state change that must be protected
    completedTasks.add(task);
} finally {
    lock.unlock();
}

Avoid locking slow work

// Usually avoid this pattern:
lock.lock();
try {
    downloadLargeFile();
    updateSmallList();
} finally {
    lock.unlock();
}
Holding a lock during slow I/O can block every other thread that needs the same lock.

Streams as an alternative to explicit threads

Sometimes developers reach for threads when what they really need is a clean data transformation pipeline. Streams help express what should happen to data, without manually managing threads.

🌊

Stream pipeline style

List<String> highPriority = tasks.stream()
    .filter(t -> t.priority() == Priority.HIGH)
    .map(Task::description)
    .toList();

Best when you are transforming, filtering, grouping, or summarizing a collection.

⚙️

Parallel stream style

long count = records.parallelStream()
    .filter(this::isValid)
    .count();

Best only when operations are independent, stateless, CPU-heavy, and the collection is large enough.

Important: a stream is not a replacement for every threaded design. Streams are strongest for data processing. Explicit threads are stronger when you need lifecycle control, blocking queues, coordination, cancellation, or long-running background services.
